    4.3 Projektdateien und .gitignore Konfiguration¶

    Einführung¶

    Die richtige Verwaltung von Projektdateien ist entscheidend für die Effizienz und Sicherheit Ihrer Projekte. In diesem Kapitel lernen Sie, welche Dateien versioniert werden sollten und welche nicht.

    Warum .gitignore wichtig ist¶

    Problemstellung¶

    Ohne .gitignore: 

    1

    Repository enthält:
- Quellcode (✓)
- Konfigurationsdateien (✓)
- Abhängigkeiten (✗)
- Build-Dateien (✗)
- Log-Dateien (✗)
- Sensible Daten (✗✗✗)

    Folgen: - Repository-Größe: Unnötige Dateien vergrößern das Repository - Sicherheitsrisiken: Sensible Daten werden öffentlich - Konflikte: Unterschiedliche Entwickler haben unterschiedliche Konfigurationen - Performance: Langsames Klonen und Arbeiten

    Was sollte versioniert werden?¶

    1. Quellcode¶

    Ja: - .py, .java, .cs, .js, .ts, .html, .css, etc. - Skripte und Konfigurationssprachen - Dokumentationsdateien (.md, .rst)

    Beispiel:

├── utils.py
├── config.yaml
└── README.md

    2. Projekt-Konfiguration¶

    Ja: - package.json (Node.js) - requirements.txt (Python) - pom.xml (Java Maven) - csproj (C#) - .editorconfig (Editor-Einstellungen) - tsconfig.json (TypeScript)

    Beispiel: 

    1

├── requirements.txt      # Python Abhängigkeiten
├── .editorconfig         # Editor-Einstellungen
└── README.md             # Projekt-Dokumentation

    3. Build-Konfiguration¶

    Ja: - Dockerfile - docker-compose.yml - .github/workflows/ (GitHub Actions) - .gitlab-ci.yml (GitLab CI) - Jenkinsfile

    Beispiel: 

    1

    4. Dokumentation¶

    Ja: - docs/ Ordner - CHANGELOG.md - CONTRIBUTING.md - LICENSE

    Was sollte NICHT versioniert werden?¶

    1. Abhängigkeiten¶

    Nein: - node_modules/ (Node.js) - venv/ (Python Virtual Environment) - __pycache__/ (Python Bytecode) - target/ (Java Maven Build) - bin/ (C# Build Output)

    Warum nicht? - Werden durch Konfigurationsdateien (package.json, requirements.txt) reproduziert - Können sehr groß sein (hunderte MB) - Plattformabhängig

    2. Build-Artefakte¶

    Nein: - dist/, build/, out/ - *.class, *.jar (Java) - *.exe, *.dll (C#) - *.pyc (Python)

    Warum nicht? - Werden bei jedem Build neu generiert - Können Konflikte verursachen - Vergrößern das Repository unnötig

    3. Konfigurationsdateien mit sensiblen Daten¶

    Nein: - .env (Umgebungsvariablen) - config.json mit Passwörtern - database.yml mit DB-Zugangsdaten - secrets.yaml

    Warum nicht? - Sicherheitsrisiko: Passwörter werden öffentlich - Compliance-Verletzung (GDPR, etc.) - Unterschiedliche Umgebungen benötigen unterschiedliche Konfigurationen

    4. Log-Dateien und temporäre Dateien¶

    Nein: - *.log - *.tmp - .DS_Store (macOS) - Thumbs.db (Windows) - *.swp, *.swo (Vim)

    5. IDE-spezifische Dateien¶

    Nein: - .vscode/ (VS Code) - .idea/ (IntelliJ) - *.suo, *.user (Visual Studio)

    Dateien und ihre Zwecke¶

    Datei/Ordner Zweck Versionieren?
    .gitignore Ignorierte Dateien ✓ Ja
    README.md Projektbeschreibung ✓ Ja
    LICENSE Software-Lizenz ✓ Ja
    CHANGELOG.md Änderungshistorie ✓ Ja
    CONTRIBUTING.md Contributing Guidelines ✓ Ja
    .env.example Umgebungsvariablen-Vorlage ✓ Ja
    .env Sensible Umgebungsvariablen ✗ Nein
    src/ Quellcode ✓ Ja
    tests/ Testcode ✓ Ja
    node_modules/ Abhängigkeiten ✗ Nein
    venv/ Virtuelle Umgebung ✗ Nein
    dist/ Build-Artefakte ✗ Nein
    .vscode/ IDE-Konfiguration ✗ Nein

    Zusammenfassung¶

    Was versionieren: - Quellcode - Projekt-Konfiguration - Build-Konfiguration - Dokumentation

    Was nicht versionieren: - Abhängigkeiten (node_modules, venv) - Build-Artefakte (dist, build) - Sensible Daten (.env, Passwörter) - Log-Dateien - IDE-Konfiguration

    .gitignore Best Practices: - Verwende Vorlagen für Ihre Sprache - Erstelle globale .gitignore für OS/IDE - Verwende .env.example für Umgebungsvariablen - Halte .gitignore aktuell

    Projektdateien und .gitignore

    Basierend auf Projektdateien und .gitignore Konfiguration:

    1. Erklären Sie, warum Abhängigkeiten (node_modules, venv) nicht versioniert werden sollten und wie man sie trotzdem reproduzierbar macht
    2. Beschreiben Sie, wie Sie sensible Daten wie Passwörter in einem Git-Repository sicher verwalten würden
    3. Erstellen Sie ein .gitignore für ein Python-Projekt mit Webanwendung, das Datenbankzugriff und API-Schlüssel verwendet
    \[\text{Beispiel: Ein Team entwickelt eine Webanwendung mit Datenbank und externen APIs...}\]
    Tipp

    Stellen Sie sich vor, Sie müssen ein Projekt für ein Finanzunternehmen erstellen. Welche zusätzlichen Sicherheitsmaßnahmen würden Sie ergreifen?

    Lösung
    Lösungsvideo

    1. Abhängigkeiten nicht versionieren: Warum nicht? - Größe: node_modules kann hunderte MB groß sein - Plattformabhängig: Unterschiedliche Systeme haben unterschiedliche Binärdateien - Redundanz: Werden durch Konfigurationsdateien reproduziert - Updates: Schwierig, Abhängigkeiten zu aktualisieren

    Reproduzierbarkeit durch: - package.json (Node.js) oder requirements.txt (Python) - package-lock.json oder Pipfile.lock für exakte Versionen - CI/CD Pipeline installiert Abhängigkeiten automatisch
