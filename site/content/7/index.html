    4.2 GitHub/GitLab Integration¶

    Einführung¶

    GitHub und GitLab sind die beiden größten Plattformen für Git-Repositories. Sie bieten weit mehr als nur Speicherplatz - sie sind komplette Entwicklungsumgebungen.

    GitHub vs. GitLab¶

    GitHub¶

    Eigenschaften: - Größte Plattform: Über 100 Millionen Entwickler - Microsoft-Eigentum: Seit 2018 - Fokus: Open Source, Community - Kosten: Kostenlos für öffentliche Repositories, private mit Einschränkungen

    Besonderheiten: - GitHub Actions (CI/CD) - GitHub Pages (Hosting) - GitHub Copilot (KI-Assistent) - GitHub Discussions - GitHub Marketplace

    GitLab¶

    Eigenschaften: - Selbst gehostet möglich: On-Premise Installation - DevOps-Plattform: Komplette CI/CD Pipeline - Fokus: Enterprise, DevOps - Kosten: Kostenlos für öffentliche Repositories, Enterprise-Lizenzen

    Besonderheiten: - Integrierte CI/CD (.gitlab-ci.yml) - Container Registry - Kubernetes Integration - Security Scanning - Self-Hosted Option

    GitHub einrichten¶

    Konto erstellen¶

    1. Besuchen: https://github.com
    2. Registrieren: E-Mail, Passwort, Benutzername
    3. E-Mail bestätigen: Link in Bestätigungsmail klicken
    4. Profil vervollständigen: Name, Bio, Avatar

    GitHub Pages¶

    Was ist GitHub Pages?¶

    Definition: Kostloser Hosting-Service für statische Websites

    Anwendungsfälle: - Dokumentation - Portfolio - Projekt-Websites - Blogs

    Einrichtung¶

    Methode 1: Repository als GitHub Pages

    1. Repository erstellen: username.github.io
    2. Index.html erstellen
    3. Push zum main Branch
    4. Website ist unter https://username.github.io verfügbar

    Methode 2: Dokumentation in Repository

    1. Dokumentation in /docs Ordner
    2. GitHub Settings → Pages
    3. Source: main Branch, /docs Ordner
    4. Website ist unter https://username.github.io/repo-name verfügbar

    GitHub vs. GitLab: Vergleich¶

    Feature GitHub GitLab
    CI/CD GitHub Actions Integriert (.gitlab-ci.yml)
    Hosting Nur Cloud Cloud + Self-Hosted
    Kosten Kostenlos (öffentlich) Kostenlos (öffentlich)
    Enterprise GitHub Enterprise GitLab Enterprise
    Dokumentation GitHub Pages GitLab Pages
    Container GitHub Container Registry GitLab Container Registry
    Security Dependabot, CodeQL Integrierte Security Scanning

    3. Branch Protection¶

    GitHub: 1. Settings → Branches → Add rule 2. Branch name pattern: main 3. Require pull request reviews 4. Require status checks 5. Require conversation resolution

    GitLab: 1. Settings → Repository → Protected branches 2. Protect main branch 3. Allow push only for maintainers 4. Require merge requests

    Zusammenfassung¶

    GitHub vs. GitLab: - GitHub: Community, Open Source, GitHub Actions - GitLab: DevOps, Self-Hosted, integrierte CI/CD

    GitHub Features: - SSH-Schlüssel für sichere Authentifizierung - GitHub Actions für Automatisierung - GitHub Pages für Hosting - Branch Protection für Sicherheit

    Best Practices: - Gute README.md - Richtiges .gitignore - Branch Protection - CI/CD Pipelines

    Tipp

    Stellen Sie sich vor, Sie müssten eine Plattform für ein Unternehmen mit 500 Entwicklern wählen. Welche Faktoren würden Sie berücksichtigen?

    Lösung
    Lösungsvideo

    1. GitHub vs. GitLab: GitHub: - Stärken: Große Community, Open Source Fokus, GitHub Actions, GitHub Pages - Anwendungsfälle: Open Source Projekte, Community-Entwicklung, schnelle Prototypen - Beispiel: Ein öffentliches JavaScript-Framework, das von der Community mitentwickelt wird

    GitLab: - Stärken: Selbst-Hosted möglich, integrierte CI/CD, Enterprise-Fokus, DevOps-Plattform - Anwendungsfälle: Unternehmensprojekte, Sicherheitskritische Anwendungen, On-Premise-Lösungen - Beispiel: Ein Finanzunternehmen, das seine Codebasis intern hosten muss

    Entscheidungshilfe: - Öffentliches Projekt → GitHub - Unternehmensintern → GitLab - Open Source Community → GitHub - Compliance-Anforderungen → GitLab

    2. SSH vs. HTTPS: SSH einrichten: 

    1
2
3
4
    1. Schlüssel generieren: ssh-keygen -t ed25519 -C "email@beispiel.de"
2. Öffentlichen Schlüssel anzeigen: cat ~/.ssh/id_ed25519.pub
3. Zu GitHub/GitLab hinzufügen
4. Testen: ssh -T git@github.com

    Vorteile von SSH: - Sicherheit: Keine Passwörter in Klartext - Komfort: Kein Passwort bei jedem Push - Automatisierung: Besser für CI/CD-Pipelines - Zugriffskontrolle: Mehrere Schlüssel pro Konto möglich

    3. CI/CD-Pipeline: Funktionsweise: 1. Trigger: Push oder Pull Request 2. Checkout: Code vom Repository holen 3. Setup: Abhängigkeiten installieren 4. Test: Automatisierte Tests ausführen 5. Build: Code kompilieren/bauen 6. Deploy: Anwendung bereitstellen
