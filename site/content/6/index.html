    Was ist ein Remote-Repository?¶

    Definition: Ein Repository, das sich auf einem entfernten Server befindet (z.B. GitHub, GitLab, Bitbucket)

    Zweck: - Zentrale Anlaufstelle für Teams - Backup der gesamten Projektgeschichte - Kollaboration über Entfernungen - Integration mit CI/CD-Tools

    Unterschied zum lokalen Repository: 

    1
2
3
4
5
    Lokales Repository:        Remote Repository:
- Auf Ihrem Computer       - Auf einem Server
- Vollständige Historie    - Vollständige Historie
- Schnelle lokale Operationen - Netzwerk-Operationen
- Keine Internetverbindung nötig - Internetverbindung nötig

    Konflikt vermeiden¶

    Best Practices: - Häufig synchronisieren: git pull regelmäßig - Kleine Commits: Kleinere Änderungen, weniger Konflikte - Kommunikation: Team über Änderungen informieren - Code Review: Änderungen vor dem Merge überprüfen

    Zusammenfassung¶

    Remote-Repositories: - Zentrale Anlaufstelle für Teams - Backup und Kollaboration - GitHub, GitLab, Bitbucket

    Synchronisation: - git push: Lokale Änderungen zum Remote - git pull: Remote-Änderungen holen und mergen - git fetch: Remote-Änderungen holen (ohne Merge)

    Collaboration-Workflows: - Feature-Branch-Workflow - GitFlow - Fork-Workflow (für Open Source)

    Konfliktlösung: - Erkennen mit git status - Manuell lösen in der Datei - Als gelöst markieren mit git add

    Tipp

    Stellen Sie sich vor, Sie schreiben ein Buch mit mehreren Autoren. Wie würden Sie Konflikte vermeiden?

    Lösung
    Lösungsvideo

    1. Unterschied zwischen push, pull und fetch: git push: - Sendet lokale Änderungen zum Remote-Repository - Beispiel: Sie haben Code geschrieben und möchten ihn mit dem Team teilen - Befehl: git push origin main

    git pull: - Holt Änderungen vom Remote und führt sie mit lokalem Code zusammen - Beispiel: Sie möchten die neuesten Änderungen Ihres Kollegen haben - Befehl: git pull origin main

    git fetch: - Holt Änderungen vom Remote, führt sie aber NICHT zusammen - Beispiel: Sie möchten sehen, was geändert wurde, bevor Sie es übernehmen - Befehl: git fetch origin

    Analogie: - push: E-Mail senden - pull: E-Mail empfangen und öffnen - fetch: E-Mail empfangen, aber noch nicht lesen

    2. Feature-Branch-Workflow: Ablauf: 1. Von main Branch starten 2. Neuen Feature-Branch erstellen 3. Feature entwickeln 4. Pull Request erstellen 5. Code Review 6. Merge in main

    Vorteile gegenüber direkter Arbeit auf main: - Isolation: Änderungen sind vom Hauptcode getrennt - Sicherheit: main bleibt immer stabil - Parallelität: Mehrere Features gleichzeitig entwickeln - Rückgängig machen: Feature-Branch kann gelöscht werden, ohne main zu beeinträchtigen

    3. Merge-Konflikt entstehen und lösen: Entstehung: 

    1
2
3
4
    Zeitpunkt 1: Entwickler A ändert Zeile 10 in datei.txt
Zeitpunkt 2: Entwickler B ändert Zeile 10 in datei.txt (gleiche Zeile)
Zeitpunkt 3: Beide pushen ihre Änderungen
Zeitpunkt 4: Git kann nicht automatisch mergen → Konflikt

    Systematisches Lösen: 1. Konflikt erkennen: git status zeigt "both modified" 2. Konflikt anzeigen: git diff oder Datei öffnen 3. Markierungen verstehen: 

    1
2
3
4
5
    <<<<<<< HEAD (Ihre Änderung)
Ihre Zeile
=======
Andere Zeile (vom anderen Branch)
>>>>>>> feature/branch
    4. Entscheidung treffen: Welche Änderung ist besser? Oder kombinieren? 5. Markierungen entfernen: Nur die gewünschte Version behalten 6. Als gelöst markieren: git add datei.txt 7. Commit erstellen: git commit -m "Resolve conflict"

    Prävention: - Häufig synchronisieren (git pull) - Kleine, häufige Commits - Kommunikation im Team
