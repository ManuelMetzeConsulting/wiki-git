    # Datei aus Historie entfernen (Vorsicht!)
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch sensibel.txt' \
  --prune-empty --tag-name-filter cat -- --all

    Große Dateien finden: 

    1
2

4

    # Große Dateien im Repository finden
git rev-list --objects --all | \
  git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
  sed -n 's/^blob //p' | \
  sort --numeric-sort --key=2 | \
  cut -c 1-12,41- | \
  numfmt --field=2 --to=iec-i --suffix=B --padding=7 --round=nearest

    Best Practices Zusammenfassung¶

    1. Repository-Management¶

    • Klare Struktur mit Standard-Verzeichnissen
    • Wichtige Dateien: README, CONTRIBUTING, LICENSE
    • .gitignore für alle unnötigen Dateien

    2. Branching¶

    • Wähle passende Branching-Strategie
    • Kurzlebige Branches bevorzugen
    • main immer deployable halten

    3. Commits¶

    • Konventionelle Commits verwenden
    • Kleine, logische Commits
    • Gute Commit-Nachrichten

    4. Code Review¶

    • Pull Requests für alle Änderungen
    • Checkliste für Reviews
    • Konstruktives Feedback

    5. CI/CD¶

    • Automatisierte Tests
    • Automatisches Deployment
    • Code Coverage messen

    6. Sicherheit¶

    • Secrets nie committen
    • Zugriffskontrolle implementieren
    • Abhängigkeiten aktualisieren

    7. Wartung¶

    • Repository regelmäßig aufräumen
    • Alte Branches löschen
    • Tags für Releases verwenden

    Repository-Struktur: - Klare, standardisierte Verzeichnisse - Wichtige Dokumente (README, CONTRIBUTING, etc.) - CI/CD Integration

    Branching-Strategien: - GitFlow für komplexe Projekte - GitHub Flow für einfache Projekte - Trunk-Based für schnelle Entwicklung

    Commit-Nachrichten: - Konventionelle Commits - Klare, beschreibende Nachrichten - Referenzen zu Issues

    CI/CD: - Automatisierte Tests - Automatisches Deployment - Code Coverage

    Sicherheit: - Secrets Management - Zugriffskontrolle - Dependabot für Updates

    2. GitFlow vs. GitHub Flow: GitFlow: - Struktur: main, develop, feature/*, release/*, hotfix/* - Vorteile: Klare Trennung, gut für große Teams, Release-Management - Nachteile: Komplex, viele Branches, langsam - Anwendung: Enterprise-Projekte, komplexe Releases, große Teams

    GitHub Flow: - Struktur: main + kurze Feature-Branches - Vorteile: Einfach, schnell, kontinuierliche Integration - Nachteile: Weniger Struktur für komplexe Projekte - Anwendung: Startups, agile Teams, kontinuierliche Deployment

    Entscheidungshilfe: - GitFlow: Finanzprojekt mit quartalsweisen Releases - GitHub Flow: SaaS-Startup mit wöchentlichen Deployment

    name: CD
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Deploy to server
      env:
        SSH_PRIVATE_KEY: ${{ secrets.SSH_PRIVATE_KEY }}
      run: |
        # SSH-Verbindung und Deployment
        echo "$SSH_PRIVATE_KEY" > key.pem
        chmod 600 key.pem
        rsync -avz -e "ssh -i key.pem" ./ user@server:/app/

    Vorteile: - Automatisches Testen bei jedem Push - Code Coverage messen - Automatisches Deployment nach main - Sicherheit durch Secrets
