# 4.3 Projektdateien und .gitignore Konfiguration

## Einführung

Die richtige Verwaltung von Projektdateien ist entscheidend für die Effizienz und Sicherheit Ihrer Projekte. In diesem Kapitel lernen Sie, welche Dateien versioniert werden sollten und welche nicht.

## Warum .gitignore wichtig ist

### Problemstellung

**Ohne .gitignore**:
```
Repository enthält:
- Quellcode (✓)
- Konfigurationsdateien (✓)
- Abhängigkeiten (✗)
- Build-Dateien (✗)
- Log-Dateien (✗)
- Sensible Daten (✗✗✗)
```

**Folgen**:
- **Repository-Größe**: Unnötige Dateien vergrößern das Repository
- **Sicherheitsrisiken**: Sensible Daten werden öffentlich
- **Konflikte**: Unterschiedliche Entwickler haben unterschiedliche Konfigurationen
- **Performance**: Langsames Klonen und Arbeiten

## Was sollte versioniert werden?

### 1. Quellcode

**Ja**:
- `.py`, `.java`, `.cs`, `.js`, `.ts`, `.html`, `.css`, etc.
- Skripte und Konfigurationssprachen
- Dokumentationsdateien (`.md`, `.rst`)

**Beispiel**:
```bash
src/
├── main.py
├── utils.py
├── config.yaml
└── README.md
```

### 2. Projekt-Konfiguration

**Ja**:
- `package.json` (Node.js)
- `requirements.txt` (Python)
- `pom.xml` (Java Maven)
- `csproj` (C#)
- `.editorconfig` (Editor-Einstellungen)
- `tsconfig.json` (TypeScript)

**Beispiel**:
```bash
project/
├── package.json          # Node.js Abhängigkeiten
├── requirements.txt      # Python Abhängigkeiten
├── .editorconfig         # Editor-Einstellungen
└── README.md             # Projekt-Dokumentation
```

### 3. Build-Konfiguration

**Ja**:
- `Dockerfile`
- `docker-compose.yml`
- `.github/workflows/` (GitHub Actions)
- `.gitlab-ci.yml` (GitLab CI)
- `Jenkinsfile`

**Beispiel**:
```bash
project/
├── Dockerfile
├── docker-compose.yml
└── .github/
    └── workflows/
        └── ci.yml
```

### 4. Dokumentation

**Ja**:
- `docs/` Ordner
- `CHANGELOG.md`
- `CONTRIBUTING.md`
- `LICENSE`

## Was sollte NICHT versioniert werden?

### 1. Abhängigkeiten

**Nein**:
- `node_modules/` (Node.js)
- `venv/` (Python Virtual Environment)
- `__pycache__/` (Python Bytecode)
- `target/` (Java Maven Build)
- `bin/` (C# Build Output)

**Warum nicht?**
- Werden durch Konfigurationsdateien (`package.json`, `requirements.txt`) reproduziert
- Können sehr groß sein (hunderte MB)
- Plattformabhängig

### 2. Build-Artefakte

**Nein**:
- `dist/`, `build/`, `out/`
- `*.class`, `*.jar` (Java)
- `*.exe`, `*.dll` (C#)
- `*.pyc` (Python)

**Warum nicht?**
- Werden bei jedem Build neu generiert
- Können Konflikte verursachen
- Vergrößern das Repository unnötig

### 3. Konfigurationsdateien mit sensiblen Daten

**Nein**:
- `.env` (Umgebungsvariablen)
- `config.json` mit Passwörtern
- `database.yml` mit DB-Zugangsdaten
- `secrets.yaml`

**Warum nicht?**
- Sicherheitsrisiko: Passwörter werden öffentlich
- Compliance-Verletzung (GDPR, etc.)
- Unterschiedliche Umgebungen benötigen unterschiedliche Konfigurationen

### 4. Log-Dateien und temporäre Dateien

**Nein**:
- `*.log`
- `*.tmp`
- `.DS_Store` (macOS)
- `Thumbs.db` (Windows)
- `*.swp`, `*.swo` (Vim)

### 5. IDE-spezifische Dateien

**Nein**:
- `.vscode/` (VS Code)
- `.idea/` (IntelliJ)
- `*.suo`, `*.user` (Visual Studio)

## .gitignore erstellen

### Grundlegende Struktur

**Datei**: `.gitignore` (im Root-Verzeichnis)

```gitignore
# Kommentare beginnen mit #

# Leerzeilen werden ignoriert
# Muster mit * für Wildcards
# Muster mit / für Verzeichnisse
# Muster mit ! für Ausnahmen
```

### Muster-Syntax

**Beispiele**:
```gitignore
# Alle .log Dateien ignorieren
*.log

# Alle .tmp Dateien ignorieren
*.tmp

# Spezifisches Verzeichnis ignorieren
node_modules/

# Alle .pyc Dateien ignorieren
*.pyc

# Ausnahme: Eine bestimmte .log Datei behalten
!important.log

# Verzeichnis mit Unterordnern ignorieren
build/

# Leerzeichen am Anfang einer Zeile = Kommentar
 # Dies ist ein Kommentar
```

### Vorlagen für verschiedene Sprachen

**Python**:
```gitignore
# Bytecode
__pycache__/
*.py[cod]
*$py.class

# Virtuelle Umgebung
venv/
.env
.env.local

# IDE
.vscode/
.idea/
*.swp
*.swo

# Build
build/
dist/
*.egg-info/
```

**JavaScript/Node.js**:
```gitignore
# Dependencies
node_modules/

# Build
dist/
build/
out/

# Logs
logs/
*.log
npm-debug.log*

# Environment
.env
.env.local
.env.development.local
.env.test.local
.env.production.local

# IDE
.vscode/
.idea/
```

**Java**:
```gitignore
# Build
target/
out/
*.class
*.jar

# Logs
*.log

# IDE
.idea/
*.iml
*.ipr
*.iws

# Maven
.mvn/
```

**C#**:
```gitignore
# Build
bin/
obj/
out/
*.exe
*.dll
*.pdb

# User-specific
*.user
*.userosscache
*.suo

# Visual Studio
.vs/
*.cache
```

### Globale .gitignore

**Warum globale .gitignore?**
- Für alle Projekte auf dem Computer
- Persönliche Einstellungen
- OS-spezifische Dateien

**Erstellen**:
```bash
# Globale .gitignore erstellen
git config --global core.excludesfile ~/.gitignore_global

# Datei bearbeiten
code ~/.gitignore_global
```

**Inhalt**:
```gitignore
# OS-spezifisch
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
*.swp
*.swo

# Temporär
*.tmp
*.temp
```

## Umgebungsvariablen verwalten

### Problem: Sensible Daten

**Falsch**:
```yaml
# config.yaml
database:
  host: localhost
  port: 5432
  username: admin
  password: supersecretpassword  # ❌ SICHERHEITSRISIKO!
```

**Richtig**:
```yaml
# config.yaml
database:
  host: localhost
  port: 5432
  username: ${DB_USER}
  password: ${DB_PASSWORD}
```

### .env-Dateien

**Beispiel .env**:
```env
# .env (NIEMALS committen!)
DB_HOST=localhost
DB_PORT=5432
DB_USER=admin
DB_PASSWORD=supersecretpassword
API_KEY=abc123xyz
SECRET_KEY=mysecretkey
```

**.env.example** (committen):
```env
# .env.example
DB_HOST=localhost
DB_PORT=5432
DB_USER=your_username
DB_PASSWORD=your_password
API_KEY=your_api_key
SECRET_KEY=your_secret_key
```

**.gitignore**:
```gitignore
# Umgebungsvariablen
.env
.env.local
.env.*.local
```

### Laden von Umgebungsvariablen

**Python**:
```python
from dotenv import load_dotenv
import os

load_dotenv()  # Lädt .env Datei

db_host = os.getenv('DB_HOST')
db_password = os.getenv('DB_PASSWORD')
```

**Node.js**:
```javascript
require('dotenv').config();

const dbHost = process.env.DB_HOST;
const dbPassword = process.env.DB_PASSWORD;
```

**C#**:
```csharp
using DotNetEnv;

Env.Load(); // Lädt .env Datei

var dbHost = Environment.GetEnvironmentVariable("DB_HOST");
var dbPassword = Environment.GetEnvironmentVariable("DB_PASSWORD");
```

## Projektstruktur best practices

### Standard-Struktur

```
mein-projekt/
├── .git/
├── .gitignore
├── README.md
├── LICENSE
├── CHANGELOG.md
├── CONTRIBUTING.md
├── .env.example
├── .github/
│   └── workflows/
│       └── ci.yml
├── docs/
│   ├── index.md
│   └── api.md
├── src/
│   ├── main.py
│   └── utils.py
├── tests/
│   ├── test_main.py
│   └── test_utils.py
├── scripts/
│   ├── setup.sh
│   └── deploy.sh
├── docker/
│   ├── Dockerfile
│   └── docker-compose.yml
└── requirements.txt
```

### Dateien und ihre Zwecke

| Datei/Ordner | Zweck | Versionieren? |
|--------------|-------|---------------|
| `.gitignore` | Ignorierte Dateien | ✓ Ja |
| `README.md` | Projektbeschreibung | ✓ Ja |
| `LICENSE` | Software-Lizenz | ✓ Ja |
| `CHANGELOG.md` | Änderungshistorie | ✓ Ja |
| `CONTRIBUTING.md` | Contributing Guidelines | ✓ Ja |
| `.env.example` | Umgebungsvariablen-Vorlage | ✓ Ja |
| `.env` | Sensible Umgebungsvariablen | ✗ Nein |
| `src/` | Quellcode | ✓ Ja |
| `tests/` | Testcode | ✓ Ja |
| `node_modules/` | Abhängigkeiten | ✗ Nein |
| `venv/` | Virtuelle Umgebung | ✗ Nein |
| `dist/` | Build-Artefakte | ✗ Nein |
| `.vscode/` | IDE-Konfiguration | ✗ Nein |

## Praktische Übung

### Übung 1: .gitignore erstellen

```bash
# 1. Projekt erstellen
mkdir gitignore-uebung
cd gitignore-uebung
git init

# 2. .gitignore erstellen
cat > .gitignore << 'EOF'
# Python
__pycache__/
*.py[cod]
*$py.class
venv/

# Node.js
node_modules/
npm-debug.log
yarn-error.log

# IDE
.vscode/
.idea/
*.swp
*.swo

# OS
.DS_Store
Thumbs.db

# Build
dist/
build/
*.log
EOF

# 3. Testdateien erstellen
mkdir src tests node_modules venv
touch src/main.py
touch tests/test_main.py
touch node_modules/package.json
touch venv/bin/activate
touch debug.log
touch .vscode/settings.json

# 4. Status prüfen
git status
# Nur src/ und tests/ sollten sichtbar sein
```

### Übung 2: Umgebungsvariablen verwalten

```bash
# 1. .env.example erstellen
cat > .env.example << 'EOF'
DB_HOST=localhost
DB_PORT=5432
DB_USER=your_username
DB_PASSWORD=your_password
API_KEY=your_api_key
EOF

# 2. .env erstellen (nicht committen!)
cat > .env << 'EOF'
DB_HOST=localhost
DB_PORT=5432
DB_USER=admin
DB_PASSWORD=supersecret
API_KEY=abc123
EOF

# 3. .gitignore aktualisieren
echo ".env" >> .gitignore

# 4. Commit
git add .
git commit -m "Add project structure and configuration"
```

### Übung 3: Globale .gitignore

```bash
# 1. Globale .gitignore erstellen
git config --global core.excludesfile ~/.gitignore_global

# 2. Datei bearbeiten
cat > ~/.gitignore_global << 'EOF'
# OS
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
*.swp
*.swo

# Temporär
*.tmp
*.temp
EOF

# 3. Testen
mkdir test-project
cd test-project
git init
touch test.tmp
touch .vscode/settings.json
git status
# Diese Dateien sollten nicht angezeigt werden
```

## Zusammenfassung

**Was versionieren**:
- Quellcode
- Projekt-Konfiguration
- Build-Konfiguration
- Dokumentation

**Was nicht versionieren**:
- Abhängigkeiten (node_modules, venv)
- Build-Artefakte (dist, build)
- Sensible Daten (.env, Passwörter)
- Log-Dateien
- IDE-Konfiguration

**.gitignore Best Practices**:
- Verwende Vorlagen für Ihre Sprache
- Erstelle globale .gitignore für OS/IDE
- Verwende .env.example für Umgebungsvariablen
- Halte .gitignore aktuell
